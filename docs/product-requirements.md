# Система баллов тьютеров

## 1. Назначение документа

Документ описывает требования к Telegram-боту для начисления, учета и обмена
баллов тьютеров. Он служит основой для проектирования, разработки, тестирования
и последующего развития приложения.

## 2. Контекст и проблематика

Тьютеры помогают ученикам на курсе программирования и получают за выполненные
действия внутренние баллы. Баллы можно обменять на привилегии школы, например:

- бесплатный курс;
- бесплатный карьерный трек;
- консультацию со старшим ментором для развития технических навыков.

Сейчас учет ведется в Яндекс Таблице. Каждый тьютер самостоятельно открывает
таблицу и записывает начисления, хотя основная рабочая коммуникация происходит
в общем Telegram-чате.

Текущий процесс создает следующие проблемы:

- для начисления баллов нужно покидать основной рабочий контекст;
- таблица неудобна для использования с телефона;
- пользователю приходится самостоятельно искать нужное действие и количество
  баллов;
- баланс, история операций и доступные привилегии не собраны в одном интерфейсе;
- заявки на привилегии и их исполнение не имеют формализованного процесса;
- изменение структуры таблицы или случайное редактирование ячеек может нарушить
  данные;
- таблица не обеспечивает явного разграничения прав тьютеров и администраторов.

## 3. Продуктовая гипотеза

Если перенести начисление и обмен баллов в Telegram-бота, тьютеры смогут
фиксировать выполненные действия непосредственно в рабочем чате, а
администраторы получат единый процесс обработки заявок. Это сократит количество
действий, снизит число ошибок и сделает состояние баланса прозрачным для
пользователя.

## 4. Цели

- Упростить самостоятельное начисление баллов тьютерами.
- Исключить ручной ввод количества баллов.
- Предоставить каждому тьютеру актуальный баланс и историю операций.
- Создать понятный процесс обмена баллов на привилегии.
- Предоставить администраторам инструменты для управления справочниками и
  заявками.
- Сделать базу данных приложения единственным источником истины.

## 5. Не входит в цели

- Проверка факта выполнения действия тьютером.
- Обязательное подтверждение каждого начисления администратором.
- Автоматическое выявление накрутки баллов.
- Начисление зарплаты или денежных выплат.
- Постоянная синхронизация с Яндекс Таблицей.
- Полноценная отдельная веб-панель в первой версии.

Система строится на доверии. История операций нужна для прозрачности,
исправления ошибок и восстановления контекста, а не для постоянного контроля.

## 6. Роли

### 6.1. Тьютер

Тьютеров может быть несколько. Тьютер может:

- выбрать выполненное действие и начислить себе установленное количество баллов;
- посмотреть текущий, доступный и зарезервированный баланс;
- посмотреть историю своих операций;
- открыть каталог привилегий;
- создать заявку на получение привилегии;
- посмотреть состояние своих заявок;
- отменить заявку, пока ее обработка не началась.

### 6.2. Администратор

Администраторов может быть несколько. Администратор может:

- просматривать и обрабатывать заявки;
- одобрять или отклонять заявки;
- отмечать одобренные заявки исполненными;
- управлять действиями и количеством начисляемых за них баллов;
- управлять каталогом привилегий и их стоимостью;
- корректировать ошибочные операции;
- назначать и снимать роли пользователей;
- просматривать журнал административных изменений.

Права администратора назначаются конкретному Telegram-аккаунту.
Роли взаимоисключающие: администратор не может одновременно пользоваться ботом
как тьютер и начислять себе баллы.

При первом запуске пользователь самостоятельно выбирает роль тьютора или
администратора. После завершения регистрации пользователь не может самостоятельно
изменить выбранную роль.

## 7. Основные бизнес-правила

1. Каждому действию соответствует фиксированное количество баллов.
2. Тьютер не может указать произвольное количество баллов.
3. Тьютер начисляет баллы только себе.
4. Начисление не требует одобрения администратора.
5. Каждое начисление сохраняется как отдельная неизменяемая операция.
6. Ошибки исправляются компенсирующей операцией, а не удалением истории.
7. Для создания заявки доступного баланса должно хватать на выбранную
   привилегию.
8. При создании заявки стоимость привилегии резервируется.
9. Зарезервированные баллы нельзя потратить повторно.
10. После исполнения заявки резерв превращается в окончательное списание.
11. При отклонении или допустимой отмене заявки резерв освобождается.
12. Стоимость привилегии фиксируется в заявке на момент ее создания. Последующее
    изменение каталога не влияет на уже созданные заявки.
13. Изменение стоимости действия влияет только на будущие начисления.
14. Действия и привилегии с существующей историей архивируются, а не удаляются.
15. База данных приложения является единственным источником истины.
16. Комментарий или ссылка при начислении необязательны.
17. Администратор не может отменить заявку после ее одобрения.
18. Роли тьютора и администратора не могут быть назначены одному пользователю
    одновременно.
19. Единицей начисления за активность в Telegram-сообществе является один
    отдельный ответ или комментарий.
20. Единицей начисления за активность в социальных сетях является одна отдельная
    публикация или один комментарий.
21. При первой регистрации пользователь самостоятельно выбирает одну роль.
22. Самостоятельная повторная регистрация и смена роли запрещены.
23. Членства в рабочем Telegram-чате достаточно для самостоятельного выбора
    любой из двух ролей, включая роль администратора.
24. Начальный баланс каждого нового тьютора во время разработки равен 0 XP.

## 8. Пользовательские сценарии

### UC-1. Регистрация пользователя

**Участник:** тьютер или администратор.

**Предусловия:** пользователь состоит в настроенном рабочем Telegram-чате.

**Основной сценарий:**

1. Пользователь запускает бота.
2. Бот получает Telegram ID и отображаемое имя пользователя.
3. Бот проверяет членство пользователя в рабочем Telegram-чате.
4. Бот предлагает выбрать роль «Тьютер» или «Администратор».
5. Пользователь подтверждает выбранную роль.
6. Бот создает профиль и фиксирует роль пользователя.
7. Бот показывает главное меню согласно роли.

**Альтернативный сценарий:** если доступ не разрешен, бот сообщает об отказе и
не предоставляет функции учета баллов.

Если пользователь уже зарегистрирован, бот не предлагает повторно выбрать роль
и сразу показывает соответствующее главное меню.

### UC-2. Начисление баллов

**Участник:** тьютер.

**Предусловия:** пользователь активен; в справочнике есть доступные действия.

**Основной сценарий:**

1. Тьютер выбирает команду «Начислить баллы».
2. Бот показывает список активных действий и количество баллов за каждое.
3. Тьютер выбирает выполненное действие.
4. Бот предлагает при желании добавить комментарий или ссылку на контекст
   помощи.
5. Бот показывает итог и запрашивает подтверждение.
6. Тьютер подтверждает операцию.
7. Бот сохраняет начисление.
8. Бот показывает начисленную сумму и новый баланс.

**Альтернативные сценарии:**

- тьютер отменяет операцию до подтверждения;
- выбранное действие было архивировано до подтверждения — бот просит выбрать
  другое;
- повторное нажатие кнопки подтверждения не создает вторую операцию.

**Результат:** баланс тьютора увеличен, операция присутствует в истории.

### UC-3. Просмотр баланса

**Участник:** тьютер.

**Основной сценарий:**

1. Тьютер выбирает команду «Баланс».
2. Бот показывает общий, зарезервированный и доступный баланс.
3. Бот предлагает перейти к истории или каталогу привилегий.

### UC-4. Просмотр истории

**Участник:** тьютер.

**Основной сценарий:**

1. Тьютер открывает историю.
2. Бот показывает операции от новых к старым.
3. Для каждой операции отображаются дата, тип, основание, количество баллов и
   связанная заявка, если она есть.
4. Тьютер может перейти к следующим страницам истории.

### UC-5. Создание заявки на привилегию

**Участник:** тьютер.

**Предусловия:** привилегия активна; доступного баланса достаточно.

**Основной сценарий:**

1. Тьютер открывает каталог привилегий.
2. Бот показывает активные привилегии, стоимость и краткое описание.
3. Тьютер выбирает привилегию.
4. Бот показывает стоимость, текущий баланс и остаток после резервирования.
5. Тьютер подтверждает заявку.
6. Бот создает заявку в статусе «На рассмотрении».
7. Бот резервирует необходимое количество баллов.
8. Бот уведомляет администраторов о новой заявке.

**Альтернативные сценарии:**

- баланса недостаточно — заявка не создается;
- привилегия архивирована до подтверждения — заявка не создается;
- повторное подтверждение не создает вторую заявку.

### UC-6. Отмена заявки тьютером

**Участник:** тьютер.

**Предусловия:** заявка находится в статусе «На рассмотрении».

**Основной сценарий:**

1. Тьютер открывает свои заявки.
2. Выбирает заявку и команду отмены.
3. Подтверждает отмену.
4. Бот переводит заявку в статус «Отменена».
5. Зарезервированные баллы снова становятся доступными.

### UC-7. Рассмотрение заявки

**Участник:** администратор.

**Предусловия:** заявка находится в статусе «На рассмотрении».

**Основной сценарий одобрения:**

1. Администратор открывает очередь заявок.
2. Выбирает заявку.
3. Проверяет пользователя, привилегию, стоимость и комментарий.
4. Одобряет заявку.
5. Бот переводит заявку в статус «Одобрена».
6. Тьютер получает уведомление.

**Сценарий отклонения:**

1. Администратор выбирает причину или вводит комментарий.
2. Бот переводит заявку в статус «Отклонена».
3. Резерв освобождается.
4. Тьютер получает уведомление с причиной.

### UC-8. Исполнение заявки

**Участник:** администратор.

**Предусловия:** заявка находится в статусе «Одобрена», привилегия фактически
предоставлена тьютеру.

**Основной сценарий:**

1. Администратор открывает одобренную заявку.
2. Отмечает ее исполненной.
3. Бот переводит заявку в статус «Исполнена».
4. Зарезервированные баллы окончательно списываются.
5. Тьютер получает уведомление.

### UC-9. Управление действиями

**Участник:** администратор.

Администратор может создать действие, изменить его название, описание и
стоимость, а также архивировать или восстановить действие. Все изменения
сохраняются в административном журнале.

### UC-10. Управление привилегиями

**Участник:** администратор.

Администратор может создать привилегию, изменить название, описание и стоимость,
а также архивировать или восстановить ее. Изменения не пересчитывают ранее
созданные заявки.

### UC-11. Корректировка баланса

**Участник:** администратор.

**Основной сценарий:**

1. Администратор выбирает тьютора.
2. Указывает положительную или отрицательную корректировку.
3. Обязательно вводит причину.
4. Подтверждает операцию.
5. Бот сохраняет корректирующую операцию и уведомляет тьютора.

Корректировка не должна делать общий баланс отрицательным без отдельного
явного подтверждения администратора.

## 9. Статусы заявки

| Статус | Значение |
| --- | --- |
| `pending` | Заявка создана и ожидает решения администратора |
| `approved` | Заявка одобрена и ожидает фактического исполнения |
| `fulfilled` | Привилегия предоставлена, баллы окончательно списаны |
| `rejected` | Заявка отклонена, резерв освобожден |
| `cancelled` | Заявка отменена тьютером или администратором, резерв освобожден |

Допустимые переходы:

- `pending` → `approved`;
- `pending` → `rejected`;
- `pending` → `cancelled`;
- `approved` → `fulfilled`.

После перехода в статус `approved` заявка не может быть отменена через бот.
Исключительные ситуации исправляются вручную на уровне базы данных с обязательной
фиксацией причины в журнале аудита.

## 10. Функциональные требования

### 10.1. Telegram-интерфейс

- Бот должен работать в личном диалоге и поддерживать вызов из рабочего чата.
- Операции с персональными данными, балансом и заявками должны продолжаться в
  личном диалоге с ботом.
- Основные действия должны быть доступны через кнопки, без необходимости
  запоминать команды.
- Каждая критичная операция должна требовать подтверждения.
- Бот должен уведомлять пользователя о результате операции.

### 10.2. Доступ и роли

- Идентификатором пользователя служит неизменяемый Telegram ID.
- Регистрация доступна только участникам настроенного рабочего Telegram-чата.
- Для проверки членства бот должен иметь необходимые права в рабочем чате.
- Идентификатор рабочего чата задается при развертывании через переменную
  окружения и не хранится в Git.
- Один пользователь может иметь только одну роль: тьютора или администратора.
- Административные команды недоступны пользователям без соответствующей роли.
- При первой регистрации пользователь самостоятельно выбирает роль.
- Выбранная роль сохраняется и не может быть самостоятельно изменена
  пользователем.
- После первоначальной настройки пользователями и ролями управляют
  администраторы через бот.

### 10.3. Баланс и операции

- Баланс должен рассчитываться из журнала операций.
- Операции не удаляются и не редактируются после создания.
- Для защиты от повторных запросов каждая операция должна иметь уникальный ключ
  идемпотентности.
- Система должна хранить автора, время, основание и связанный объект операции.

### 10.4. Уведомления

- Администраторы получают уведомления о новых заявках.
- Тьютер получает уведомления об одобрении, отклонении, отмене и исполнении.
- Ошибка доставки уведомления не должна откатывать успешно сохраненную операцию.

## 11. Сущности данных

### User

- внутренний идентификатор;
- Telegram ID;
- имя и username;
- признак активности;
- дата создания и последнего обновления.

### Role

- пользователь;
- роль: `tutor` или `admin`;
- кто и когда назначил роль.

### Activity

- название и описание действия;
- количество начисляемых баллов;
- признак активности;
- автор и время изменений.

### PointTransaction

- пользователь;
- тип: начисление, резерв, освобождение резерва, списание или корректировка;
- количество баллов;
- действие, заявка или иное основание;
- комментарий;
- автор операции;
- ключ идемпотентности;
- дата создания.

### Reward

- название и описание привилегии;
- стоимость в баллах;
- признак активности;
- автор и время изменений.

### RewardRequest

- тьютер;
- привилегия;
- зафиксированные название и стоимость;
- статус;
- комментарий тьютора;
- решение и комментарий администратора;
- администратор, обработавший заявку;
- даты создания, одобрения, отклонения, отмены и исполнения.

### AuditLog

- администратор;
- вид административного действия;
- измененный объект;
- предыдущее и новое значения;
- дата изменения.

## 12. Нефункциональные требования

- Все изменения баланса и статуса заявки должны выполняться транзакционно.
- Одновременная обработка одной заявки двумя администраторами не должна приводить
  к двойному списанию.
- После перезапуска бот не должен терять данные или состояние подтвержденных
  операций.
- Секрет Telegram-бота и параметры базы данных не должны храниться в Git.
- Административные действия должны журналироваться.
- Пользовательские сообщения не должны раскрывать внутренние ошибки или секреты.
- Должно выполняться регулярное резервное копирование базы данных.
- Время ответа на обычную команду при штатной нагрузке — не более двух секунд,
  не считая задержек Telegram.

## 13. Первоначальное заполнение данных

Полноценная автоматическая миграция из Яндекс Таблицы не требуется. Пользователи
регистрируются самостоятельно при первом запуске бота и выбирают роль.

Seed выполняется только при первоначальном развертывании и:

1. Загружает стартовый справочник действий.
2. Загружает стартовый каталог привилегий.
3. Не создает повторные записи при повторном безопасном запуске.

Во время разработки каждый новый тьютер получает стартовый баланс 0 XP.
Персональные стартовые балансы и импорт данных не требуются. Яндекс Таблица не
используется приложением и после запуска может быть архивирована.

## 14. Этапы разработки

### Этап 0. Уточнение правил

- Проверить названия и описания стартовых действий и привилегий.
- Подготовить seed-файл справочников.
- Подготовить переменную окружения для идентификатора рабочего Telegram-чата без
  добавления реального значения в Git.

**Результат:** утвержденные справочники и готовые стартовые данные.

### Этап 1. Технический фундамент

- Настроить Telegram-бота, конфигурацию и базу данных.
- Создать модели пользователей, ролей, действий и операций.
- Реализовать проверку членства в рабочем Telegram-чате.
- Реализовать самостоятельный выбор и фиксацию роли.
- Реализовать идемпотентный seed справочников.
- Добавить миграции базы данных и базовое журналирование.

**Критерий готовности:** разрешенный пользователь запускает бота и видит меню
согласно своей роли; данные сохраняются после перезапуска.

### Этап 2. Начисление и баланс

- Реализовать каталог действий.
- Реализовать подтверждаемое начисление.
- Реализовать баланс и историю операций.
- Добавить защиту от повторного начисления при повторной доставке запроса.

**Критерий готовности:** тьютер может начислить фиксированные баллы, увидеть
обновленный баланс и найти операцию в истории.

### Этап 3. Привилегии и заявки

- Реализовать каталог привилегий.
- Реализовать резервирование баллов и создание заявки.
- Реализовать список и карточку заявок тьютора.
- Реализовать отмену заявки в допустимом статусе.

**Критерий готовности:** тьютер с достаточным балансом создает заявку, а ее
стоимость исключается из доступного баланса.

### Этап 4. Административные процессы

- Реализовать очередь заявок.
- Реализовать одобрение, отклонение, отмену и исполнение.
- Реализовать уведомления.
- Реализовать управление действиями и привилегиями.
- Реализовать корректировки баланса и аудит.

**Критерий готовности:** несколько администраторов могут безопасно обрабатывать
заявки без двойного списания, а тьютер получает уведомления.

### Этап 5. Начальное заполнение и пилот

- Подготовить и протестировать seed справочников на тестовой базе.
- Зарегистрировать участников.
- Проверить, что каждый новый тьютер начинает с баланса 0 XP.
- Провести пилот с несколькими тьюторами и администраторами.
- Исправить выявленные проблемы.
- Архивировать Яндекс Таблицу.

**Критерий готовности:** балансы сверены, основные сценарии пройдены реальными
пользователями, новые операции создаются только через бота.

### Этап 6. Эксплуатация и развитие

- Настроить мониторинг, резервное копирование и восстановление.
- Добавить отчеты по запросу команды.
- Улучшать навигацию на основе обратной связи.
- Рассмотреть напоминания, экспорт и аналитику после стабилизации MVP.

## 15. Критерии успеха

- Большинство начислений создается через Telegram без обращения к внешним
  таблицам.
- Тьютер может выполнить начисление не более чем за несколько последовательных
  действий.
- Количество баллов невозможно случайно ввести вручную.
- Баланс и история доступны тьютеру в любой момент.
- Каждая заявка имеет понятный статус и ответственного администратора.
- Ни одна заявка не приводит к двойному списанию.
- После начального заполнения нет расхождений между подтвержденными балансами и
  данными приложения.

## 16. Риски и способы снижения

| Риск | Мера |
| --- | --- |
| Ошибочное или повторное начисление | Подтверждение, идемпотентность и административная корректировка |
| Потеря данных | Транзакции, резервные копии и проверка восстановления |
| Одновременная обработка заявки | Блокировка или атомарная проверка статуса в базе |
| Потеря доступа к Telegram-аккаунту администратора | Несколько администраторов и процедура переназначения роли |
| Получение административных прав обычным участником | Это принятое ограничение доверительной модели; выбор роли фиксируется в журнале |
| Устаревшие действия или привилегии | Архивация справочников без удаления истории |
| Недоверие к итоговому балансу | Детальная история операций и аудит изменений |
| Ошибки начального заполнения | Тестовый запуск seed, идемпотентность и ручная сверка |

Telegram-чата будет предоставлен только при настройке окружения и не должен
попадать в документацию, исходный код или Git.
