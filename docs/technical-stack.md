# Технический стек и каркас приложения

## 1. Контекст

Приложение предназначено для небольшой команды: около пяти тьютеров и пяти
администраторов. На первом этапе бот запускается одним процессом и не требует
архитектуры, рассчитанной на высокую нагрузку.

Главные технические требования связаны не с производительностью, а с
корректностью учета баллов:

- сохранение данных после перезапуска;
- транзакционное резервирование и списание баллов;
- защита от повторной обработки Telegram-событий;
- безопасная одновременная обработка заявки администраторами;
- хранение неизменяемой истории операций и административного аудита.

## 2. Выбранные технологии

| Область | Технология | Назначение |
| --- | --- | --- |
| Среда выполнения | Bun | Запуск приложения, управление зависимостями и выполнение тестов |
| Язык | TypeScript | Основной язык приложения и статическая проверка типов |
| Telegram Bot API | grammY | Обработка команд, сообщений и inline-кнопок |
| База данных | PostgreSQL | Постоянное хранение данных, транзакции и ограничения целостности |
| Доступ к базе данных | Drizzle ORM | Описание схемы, типизированные запросы и транзакции |
| Миграции | Drizzle Kit | Создание и применение миграций PostgreSQL |
| Тестирование | `bun:test` | Модульные и интеграционные тесты |
| Статический анализ | ESLint | Проверка качества и стиля TypeScript-кода |

## 3. Способ запуска

В первой версии бот работает через Telegram long polling:

- запускается один экземпляр приложения;
- публичный HTTP-сервер и webhook не требуются;
- состояние подтвержденных операций хранится только в PostgreSQL;
- временное состояние диалога не должно быть единственным источником данных для
  бизнес-операций.

Приложение может быть развернуто как один процесс или один контейнер рядом с
доступным экземпляром PostgreSQL.

## 4. Каркас проекта

```text
src/
  config.ts
  index.ts
  bot.ts
  db/
    index.ts
    schemes/
      users.ts
  handlers/
    index.ts
    registration.ts
  services/
    index.ts
    findUser.ts
    registerUser.ts
  keyboards/
    roleKeyboard.ts
  domains/
    users.ts
  types/
    bot.ts
```

Назначение основных частей:

- `handlers` принимают Telegram-события, проверяют пользовательский ввод и
  формируют ответы;
- `services` содержат бизнес-правила и границы транзакций;
- `db` содержит схему, миграции, подключение к PostgreSQL и начальные данные;
- `keyboards` содержит Telegram-клавиатуры и callback-идентификаторы;
- `config.ts` читает и проверяет обязательные переменные окружения.

По мере роста проекта каталоги могут делиться по предметным областям. На старте
дополнительные архитектурные слои не вводятся.

## 5. Обязательные технические решения

- Создание заявки и резервирование баллов выполняются в одной транзакции.
- Изменение статуса заявки и связанные операции с баллами выполняются в одной
  транзакции.
- Повторное подтверждение защищается уникальным ключом идемпотентности в базе.
- Конкурентная обработка заявки защищается атомарной проверкой текущего статуса
  или блокировкой строки PostgreSQL.
- Баланс рассчитывается из журнала операций, а не хранится как независимо
  редактируемое значение.
- Уведомление отправляется после успешной фиксации транзакции. Ошибка Telegram
  не отменяет сохраненную бизнес-операцию.
- Технические ошибки на старте записываются стандартными средствами
  `console.error`.
- Административный `AuditLog` хранится в PostgreSQL, поскольку является частью
  бизнес-требований, а не технической системой логирования.

## 6. Конфигурация

Минимальный набор переменных окружения:

```text
BOT_API_KEY=
DATABASE_URL=
WORK_CHAT_ID=
```

Секреты и реальные значения переменных окружения не добавляются в Git. В
репозитории хранится только пример конфигурации без секретных значений.

## 7. Не используем на старте

Пока требования и нагрузка не изменятся, проекту не нужны:

- отдельный веб-фреймворк или REST API;
- Redis;
- брокер сообщений и отдельная очередь задач;
- микросервисы;
- Kubernetes;
- отдельная административная веб-панель;
- Sentry и специализированная система технического логирования;
- отдельный тестовый фреймворк поверх `bun:test`.

Эти технологии добавляются только при появлении конкретной задачи, которую
нельзя разумно решить текущим стеком.
